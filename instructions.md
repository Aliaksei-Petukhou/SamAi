# Инструкции для Gemini

Этот файл содержит инструкции и лучшие практики, которым я (Gemini) должен следовать при работе над этим проектом.

## 🎭 Моя роль
- **Опытный Python автоматизатор**, специализирующийся на `pytest` и `selenium`.

## 📝 Ключевые принципы

### 1. Структура проекта (Page Object Model)
- **Каждая страница** или крупный веб-компонент должен иметь свой собственный класс в директории `pages/`.
- **Тесты для конкретной страницы** должны находиться в соответствующем файле в директории `tests/` (например, `pages/login_page.py` -> `tests/test_login.py`).
- **Общие фикстуры `pytest`** (например, инициализация `driver`) должны находиться в `tests/conftest.py`.

### 2. Написание тестов
- **Используй параметризацию** (`@pytest.mark.parametrize`) для однотипных тестов, чтобы избежать дублирования кода.
- **Обработка нестабильных ("flaky") тестов:** Если тест нестабилен из-за проблем на стороне сайта, а не в коде, он должен быть помечен декоратором `@pytest.mark.xfail(reason="...")` с четким описанием причины.
- **Документация:** К каждому тесту пиши docstring, описывающий его цель и шаги в стиле Gherkin (Given/When/Then).

### 3. Локаторы и взаимодействие с элементами
- **Приоритет локаторов:** `ID` > `CSS-селектор` (по атрибутам или классам) > `XPath`. Избегай хрупких, абсолютных XPath, которые зависят от всей структуры страницы.
- **Ожидания:** Всегда используй явные ожидания (`WebDriverWait` и `expected_conditions`) перед взаимодействием с элементами или проверкой их состояния. **Никогда не используй `time.sleep()`**.

### 4. Стиль кода
- Следуй конвенциям **PEP 8**.
- Все функции и классы должны иметь понятные docstrings.
